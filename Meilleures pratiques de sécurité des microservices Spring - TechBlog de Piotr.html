Meilleures pratiques de sécurité de Spring Microservices

Meilleures pratiques de sécurité de Spring Microservices

Par piotr.minkowski 26 mai 2021 dix

Dans cet article, je décrirai plusieurs bonnes pratiques pour créer des microservices avec Spring Boot et Spring Security. Je vais me concentrer uniquement sur les aspects liés à la sécurité. Si vous êtes intéressé par la liste générale des meilleures pratiques pour créer des microservices avec Spring Boot, lisez mon article Spring Boot Best Practices for Microservices . En revanche, si vous envisagez d'exécuter vos applications sur Kubernetes, vous pourriez être intéressé par l'article Best Practices For Microservices on Kubernetes .

Avant de commencer avec une liste de « règles d'or » de sécurité, analysons une architecture de microservices typique. Nous nous concentrerons sur les composants importants pour la construction d'une solution sécurisée.

L'image visible ci-dessous illustre une architecture de microservices typique construite avec Spring Cloud. Il existe une passerelle API construite au-dessus de Spring Cloud Gateway. Puisqu'il s'agit d'un point d'entrée dans notre système, nous y activerons certains mécanismes de sécurité importants. Il y a plusieurs microservices cachés derrière la passerelle. Il existe également un serveur de découverte, qui permet de localiser les adresses IP en utilisant le nom des services. Et enfin, il y a certains composants qui ne participent pas directement à la communication. Il ne s'agit que d'une proposition de quelques outils sélectionnés. Vous pouvez choisir d'autres solutions offrant les mêmes fonctionnalités. Vault est un outil pour stocker et accéder en toute sécurité aux secrets. Keycloak est une solution open source de gestion des identités et des accès. Spring Cloud Config Server fournit une API HTTP pour la configuration externe.

Commençons. Voici notre liste des meilleures pratiques de Spring Security.

1. Activer la limitation de débit sur la passerelle API

Une passerelle API est un modèle important dans l'architecture basée sur les microservices. Il agit comme un point d'entrée unique dans l'ensemble du système. Il est responsable non seulement du routage des demandes, mais également de plusieurs autres éléments, dont la sécurité. Par conséquent, l'un des composants les plus essentiels de la sécurité que nous devrions activer sur la passerelle est la limitation du débit. Il protège votre API contre les attaques DoS, qui peuvent tanker un serveur avec des requêtes HTTP illimitées.

Spring fournit sa propre implémentation du modèle de passerelle API appelé Spring Cloud Gateway. D'autre part, Spring Cloud Gateway est livré avec une implémentation intégrée d'un composant de limitation de débit. Pour résumer, il vous suffit d'inclure une seule dépendance pour créer une application de passerelle. Ensuite, vous devez fournir certains paramètres de configuration pour activer la limitation de débit pour une seule route.

Afin d'activer un limiteur de débit sur une passerelle, nous devons utiliser un composant appelé RequestRateLimiter GatewayFilter factory. Il utilise une  RateLimiter implémentation pour déterminer si la demande en cours est autorisée à continuer. Sinon, la passerelle renvoie un état de HTTP 429 - Too Many Requests. L' RequestRateLimiterimplémentation utilise Redis comme backend.

Examinons une configuration typique de routes gérées par Spring Cloud Gateway. Trois paramètres peuvent être utilisés pour configurer le limiteur de débit : replenishRate, burstCapacityet requestedTokens. La  replenishRate propriété est le nombre de requêtes par seconde qu'un seul utilisateur peut envoyer. La  burstCapacity propriété est le nombre maximum de requêtes qu'un utilisateur peut envoyer en une seule seconde. Avec la requestedTokenspropriété, nous pouvons définir le coût d'un seul jeton.

spring:
  cloud:
    gateway:
      routes:
        - id: account-service
          uri: http://localhost:8090
          predicates:
            - Path=/account/**
          filters:
            - RewritePath=/account/(?<path>.*), /$\{path}
            - name: RequestRateLimiter
              args:
                redis-rate-limiter.replenishRate: 1
                redis-rate-limiter.burstCapacity: 60
                redis-rate-limiter.requestedTokens: 15

Ensuite, nous devrions définir un KeyResolverbean. Un limiteur de débit définit tous les paramètres par une seule clé renvoyée par le résolveur.

@Bean
KeyResolver authUserKeyResolver() {
   return exchange -> ReactiveSecurityContextHolder.getContext()
            .map(ctx -> ctx.getAuthentication().getPrincipal().toString());
}

Pour plus de détails, vous pouvez vous référer à l'article Secure Rate Limiting with Spring Cloud Gateway .

2. Générer et propager dynamiquement des certificats

Devrions-nous utiliser SSL dans la communication microservice à microservice ? Bien sûr que oui. Mais la question est de savoir comment allez-vous gérer les certificats utilisés par vos microservices. Il existe plusieurs bonnes pratiques liées à la gestion des certificats SSL. Par exemple, vous ne devez pas émettre de certificats pour de longues périodes. Vous devez également les renouveler ou les actualiser automatiquement.

Certains outils peuvent aider à suivre les meilleures pratiques. L'un des plus populaires d'entre eux est Vault de Hashicorp. Il fournit le moteur de secrets PKI, qui est chargé de générer des certificats X.509 dynamiques. Le moyen le plus simple d'essayer Vault est de l'exécuter localement sur un conteneur Docker.

$ docker run --cap-add=IPC_LOCK -d --name vault -p 8200:8200 vault

Ensuite, nous pouvons activer et configurer le moteur PKI en quelques étapes simples. Vous pouvez le faire à l'aide de Vault CLI ou de l'interface utilisateur. Pour des informations plus détaillées à ce sujet, lisez mon article SSL avec Spring WebFlux et Vault PKI . Pour l'instant, activons PKI avec TTL, puis configurons CA à l'aide de CLI comme indiqué ci-dessous.

$ vault secrets enable pki
$ vault secrets tune -max-lease-ttl=8760h pki
$ vault write pki/root/generate/internal \
    common_name=my-website.com \
    ttl=8760h

Dans l'étape suivante, nous utiliserons Spring VaultTemplatepour émettre un certificat de manière dynamique. Le fragment de code visible ci-dessous montre comment créer une demande de certificat avec 12h TTL et localhosten tant que Common Name. Tout d'abord, construisons une telle requête en utilisant l' VaultCertificateRequestobjet. Ensuite, nous invoquerons la issueCertificateméthode sur l' VaultPkiOperationsobjet. Le généré CertificateBundlecontient à la fois le certificat et la clé privée.

private CertificateBundle issueCertificate() throws Exception {
   VaultPkiOperations pkiOperations = vaultTemplate.opsForPki("pki");
   VaultCertificateRequest request = VaultCertificateRequest.builder()
        .ttl(Duration.ofHours(12))
        .commonName("localhost")
        .build();
   VaultCertificateResponse response = pkiOperations.issueCertificate("default", request);
   CertificateBundle certificateBundle = response.getRequiredData();
   return certificateBundle;
}

Enfin, il nous suffit d'utiliser la méthode de génération d'un certificat dans Vault lors de l'exécution. Le comportement par défaut de notre serveur Web doit être remplacé. Pour ce faire, nous devons créer un Spring @Componentqui implémente WebServerFactoryCustomizer. Selon le serveur Web, nous devons personnaliser un fichier WebServerFactory. En règle générale, pour Spring MVC, il s'agit de Tomcat et Netty pour Spring WebFlux. Dans la customizeméthode, nous générons un certificat et le stockons dans le keystore (cert + clé privée) et le truststore (CA).

@Component
public class GatewayServerCustomizer implements 
         WebServerFactoryCustomizer<NettyReactiveWebServerFactory> {

   @SneakyThrows
   @Override
   public void customize(NettyReactiveWebServerFactory factory) {
      String keyAlias = "vault";
      CertificateBundle bundle = issueCertificate();
      KeyStore keyStore = bundle.createKeyStore(keyAlias);
      String keyStorePath = saveKeyStoreToFile("server-key.pkcs12", keyStore);

      Ssl ssl = new Ssl();
      ssl.setEnabled(true);
      ssl.setClientAuth(Ssl.ClientAuth.NEED);

      ssl.setKeyStore(keyStorePath);
      ssl.setKeyAlias(keyAlias);
      ssl.setKeyStoreType(keyStore.getType());
      ssl.setKeyPassword("");
      ssl.setKeyStorePassword("123456");

      X509Certificate caCert = bundle.getX509IssuerCertificate();
      log.info("CA-SerialNumber: {}", caCert.getSerialNumber());
      KeyStore trustStore = KeyStore.getInstance("pkcs12");
      trustStore.load(null, null);
      trustStore.setCertificateEntry("ca", caCert);
      String trustStorePath = saveKeyStoreToFile("server-trust.pkcs12", trustStore);

      ssl.setTrustStore(trustStorePath);
      ssl.setTrustStorePassword("123456");
      ssl.setTrustStoreType(trustStore.getType());

      factory.setSsl(ssl);
      factory.setPort(8443);
   }
}

3. Utiliser SSL dans la communication des microservices

Étant donné que l'utilisation de SSL à la périphérie d'un système basé sur des microservices est évidente, la communication interservices est parfois considérée comme non sécurisée. Ma recommandation est toujours la même. Mieux vaut utiliser SSL ou pas 🙂 Mais il faudrait penser à sécuriser au moins les composants qui stockent des données sensibles. L'un d'eux sera probablement un serveur de configuration. C'est évidemment l'une des meilleures pratiques de Spring Security. Pour les microservices Spring Boot, nous pouvons utiliser un composant appelé Spring Cloud Config Server. Comme il est construit sur Spring MVC, nous pouvons facilement activer une connexion sécurisée côté serveur.

server:
  port: ${PORT:8888}
  ssl:
    enabled: true
    client-auth: need
    key-store: classpath: server-key.jks
    key-store-password: 123456
    key-alias: configserver
    trust-store: classpath: server-trust.jks
    trust-store-password: 123456

Côté client, nous utilisons un composant appelé Spring Cloud Config Client. Puisqu'il est responsable de la connexion avec le serveur, nous devons également y gérer SSL. Pour ce faire, nous devons remplacer la RestTemplateconfiguration SSL sur le ConfigServicePropertySourceLocatorbean. Le fragment de code visible ci-dessous utilise un certificat auto-signé, mais nous pouvons facilement mettre en œuvre ici une stratégie décrite dans la section précédente.

@Configuration
public class SSLConfigServiceBootstrapConfiguration {

   @Autowired
   ConfigClientProperties properties;

   @Bean
   public ConfigServicePropertySourceLocator configServicePropertySourceLocator() throws Exception {
      final char[] password = "123456".toCharArray();
      final ClassPathResource resource = new ClassPathResource("account.jks");

      SSLContext sslContext = SSLContexts.custom()
         .loadKeyMaterial(resource.getFile(), password, password)
         .loadTrustMaterial(resource.getFile(), password, new TrustSelfSignedStrategy()).build();
      CloseableHttpClient httpClient = HttpClients.custom()
         .setSSLContext(sslContext)
         .setSSLHostnameVerifier((s, sslSession) -> true)
         .build();
      HttpComponentsClientHttpRequestFactory requestFactory = new HttpComponentsClientHttpRequestFactory(httpClient);
      ConfigServicePropertySourceLocator configServicePropertySourceLocator = new ConfigServicePropertySourceLocator(properties);
      configServicePropertySourceLocator.setRestTemplate(new RestTemplate(requestFactory));
      return configServicePropertySourceLocator;
   }

}

Dans la dernière version de Spring Cloud Config, nous pouvons activer le chiffrement du trafic TLS dans la configuration. Nous avons juste besoin de définir les bons paramètres en utilisant des propriétés avec un préfixe spring.cloud.config.tls.*.

Qu'en est-il du cryptage des communications entre les applications et un serveur de découverte ? Vous pouvez choisir entre plusieurs serveurs de découverte disponibles pris en charge dans Spring Cloud. Mais supposons que nous utilisons Eureka. Comme pour Spring Cloud Config, nous utilisons un client de haut niveau pour communiquer avec un serveur. Donc, dans ce cas, nous devons définir un bean DiscoveryClientOptionalArgset également remplacer les paramètres SSL sur le client HTTP. Le client Eureka utilise le client HTTP Jersey, nous devons donc créer une instance de EurekaJerseyClientBuilderpour remplacer la configuration SSL.

@Bean
public DiscoveryClient.DiscoveryClientOptionalArgs discoveryClientOptionalArgs() throws Exception {
   DiscoveryClient.DiscoveryClientOptionalArgs args = new DiscoveryClient.DiscoveryClientOptionalArgs();
   final char[] password = "123456".toCharArray();
   final ClassPathResource resource = new ClassPathResource("account.jks");

   SSLContext sslContext = SSLContexts.custom()
	.loadKeyMaterial(resource.getFile(), password, password)
	.loadTrustMaterial(resource.getFile(), password, new TrustSelfSignedStrategy()).build();

   EurekaJerseyClientBuilder builder = new EurekaJerseyClientBuilder();	 
   builder.withClientName("account-client");
   builder.withMaxTotalConnections(10);
   builder.withMaxConnectionsPerHost(10);
   builder.withCustomSSL(sslContext);
   args.setEurekaJerseyClient(builder.build());
   return args;
}

Enfin, nous pouvons configurer HTTPS dans la communication entre les microservices. Étant donné que nous utilisons RestTemplateou WebClientinstances directement côté client, il est relativement facile de mettre en œuvre une communication sécurisée dans ce cas.

4. Gardez les données de configuration chiffrées

L'une des meilleures pratiques actuelles pour la sécurité des microservices Spring est liée à un serveur de configuration. Nous devrions chiffrer au moins les données sensibles comme les mots de passe ou les secrets qui y sont stockés. Spring Cloud Config Server fournit un mécanisme intégré pour cela. Mais nous pouvons également utiliser Vault comme magasin principal pour Spring Cloud Config Server, où toutes les données sont chiffrées par défaut.

Nous commencerons par un mécanisme de chiffrement par défaut fourni par Spring Cloud Config Server. Tout d'abord, nous devons l'activer dans les propriétés de configuration.

spring:
  cloud:
    config:
      server:
        encrypt:
          enabled: true

Ensuite, nous devons configurer un magasin de clés chargé de chiffrer nos données sensibles.

encrypt:
  keyStore:
    location: classpath:/config.jks
    password: 123456
    alias: config
    secret: 123456

Enfin, nous pouvons définir des données cryptées au lieu d'une chaîne simple avec {cipher}préfixe.

5. Restreindre l'accès aux ressources de l'API

Dans les sections précédentes, nous avons abordé des sujets tels que l'authentification, le trafic et le chiffrement des données. Mais un autre aspect important de la sécurisation de vos applications est l'autorisation et l'accès aux ressources de l'API. Si vous pensez à l'autorisation des applications Web, la première approche qui vous vient probablement à l'esprit est OAuth 2.0 ou OpenID Connect. OAuth 2.0 est le protocole standard de l'industrie pour l'autorisation. Bien sûr, il est pris en charge par Spring Security. Il existe également plusieurs fournisseurs OAuth2 avec lesquels vous pouvez intégrer votre application. L'un d'eux est Keycloak. Je vais l'utiliser dans l'exemple de cet article. Tout d'abord, exécutons Keycloak sur un conteneur Docker. Par défaut, il expose l'API et une console Web sur le port  8080.

$ docker run -d --name keycloak -p 8888:8080 \
   -e KEYCLOAK_USER=spring \
   -e KEYCLOAK_PASSWORD=spring123 \
   jboss/keycloak

Nous allons activer et configurer la prise en charge d'OAuth 2.0 sur la passerelle API. Outre spring-cloud-starter-gateway la dépendance, nous devons inclure spring-boot-starter-oauth2-client et spring-cloud-starter-security activer le  TokenRelay filtre. Ensuite, nous devons fournir les paramètres de configuration de Spring Security pour le client OAuth2.

spring:
  security:
    oauth2:
      client:
        provider:
          keycloak:
            token-uri: http://localhost:8080/auth/realms/master/protocol/openid-connect/token
            authorization-uri: http://localhost:8080/auth/realms/master/protocol/openid-connect/auth
            userinfo-uri: http://localhost:8080/auth/realms/master/protocol/openid-connect/userinfo
            user-name-attribute: preferred_username
        registration:
          keycloak-with-test-scope:
            provider: keycloak
            client-id: spring-with-test-scope
            client-secret: c6480137-1526-4c3e-aed3-295aabcb7609
            authorization-grant-type: authorization_code
            redirect-uri: "{baseUrl}/login/oauth2/code/keycloak"
          keycloak-without-test-scope:
            provider: keycloak
            client-id: spring-without-test-scope
            client-secret: f6fc369d-49ce-4132-8282-5b5d413eba23
            authorization-grant-type: authorization_code
            redirect-uri: "{baseUrl}/login/oauth2/code/keycloak"

Dans la dernière étape, nous devons configurer le filtre Spring Security. Étant donné que Spring Cloud Gateway est construit sur Spring WebFlux, nous devons annoter le bean de configuration avec  @EnableWebFluxSecurity. Dans la  filterChain méthode, nous allons activer l'autorisation pour tous les échanges. Nous allons également définir OAuth2 comme méthode de connexion par défaut et enfin désactiver CSRF.

@Configuration
@EnableWebFluxSecurity
public class SecurityConfig {

   @Bean
   public SecurityWebFilterChain filterChain(ServerHttpSecurity http) {
      http.authorizeExchange(exchanges -> exchanges.anyExchange().authenticated())
         .oauth2Login(withDefaults());
      http.csrf().disable();
      return http.build();
   }

}

Comme je l'ai mentionné précédemment, nous établirons un modèle de relais de jeton entre la passerelle et les microservices. Un relais de jeton est l'endroit où un consommateur OAuth2 agit en tant que client et transfère le jeton entrant aux demandes de ressources sortantes. Alors, activons maintenant la sécurité globale des méthodes et le serveur de ressources OAuth2 pour les services en aval.

@Configuration
@EnableWebSecurity
@EnableGlobalMethodSecurity(prePostEnabled = true)
public class SecurityConfig extends WebSecurityConfigurerAdapter {

   protected void configure(HttpSecurity http) throws Exception {
      http.authorizeRequests(authorize -> authorize.anyRequest().authenticated())
            .oauth2ResourceServer(OAuth2ResourceServerConfigurer::jwt);
   }
}

Après cela, il est possible de configurer l'accès basé sur les rôles en utilisant  @PreAuthorize et  @PostAuthorize. Examinons l'implémentation de la classe contrôleur REST. C'est une  ping méthode unique. Cette méthode n'est accessible qu'au client avec l'  TEST étendue. Pour plus de détails sur la mise en œuvre, vous pouvez vous référer à l'article Spring Cloud Gateway OAuth2 avec Keycloak .

@RestController
@RequestMapping("/callme")
public class CallmeController {

   @PreAuthorize("hasAuthority('SCOPE_TEST')")
   @GetMapping("/ping")
   public String ping() {
      SecurityContext context = SecurityContextHolder.getContext();
      Authentication authentication = context.getAuthentication();
      return "Scopes: " + authentication.getAuthorities();
   }
}

6. Générer dynamiquement des identifiants pour les systèmes externes

Votre application se connecte-t-elle à des systèmes externes tels que des bases de données ou des courtiers de messages ? Comment stockez-vous les informations d'identification utilisées par votre application ? Bien sûr, nous pouvons toujours chiffrer les données sensibles, mais si nous travaillons avec de nombreux microservices ayant des bases de données distinctes, ce n'est peut-être pas une solution très confortable. Voici Vault avec un autre mécanisme pratique. Son moteur de secrets de base de données génère dynamiquement des informations d'identification de base de données en fonction des rôles configurés. Nous pouvons également tirer parti des informations d'identification générées dynamiquement pour RabbitMQ, Nomad et Consul.

Tout d'abord, activons le moteur de base de données Vault, qui est désactivé par défaut.

$ vault secrets enable database

Supposons que notre application se connecte à la base de données Postgres. Par conséquent, nous devons configurer un plug-in Vault pour la base de données PostgreSQL, puis fournir les paramètres de connexion et les informations d'identification.

$ vault write database/config/postgres plugin_name=postgresql-database-plugin allowed_roles="default" connection_url="postgresql://{{username}}:{{password}}@localhost:5432?sslmode=disable" username="postgres" password="postgres123456"

Ensuite, nous devons créer un rôle de base de données. Le nom du rôle doit être le même que le nom passé dans le champ  allowed_roles à l'étape précédente. Nous devons également définir un nom de base de données cible et une instruction SQL qui crée des utilisateurs avec des privilèges.

$ vault write database/roles/default db_name=postgres creation_statements="CREATE ROLE \"{{name}}\" WITH LOGIN PASSWORD '{{password}}' VALID UNTIL '{{expiration}}';GRANT SELECT, UPDATE, INSERT ON ALL TABLES IN SCHEMA public TO \"{{name}}\";GRANT USAGE,  SELECT ON ALL SEQUENCES IN SCHEMA public TO \"{{name}}\";" default_ttl="1h" max_ttl="24h"

Grâce au projet Spring Cloud Vault, nous pouvons facilement intégrer n'importe quelle application Spring Boot avec le moteur de bases de données Vault. Deux dépendances doivent être incluses dans Maven pom.xmlpour activer cette prise en charge. Bien sûr, nous avons également besoin de dépendances pour le pilote JPA et Postgres.

<dependency>
   <groupId>org.springframework.cloud</groupId>
   <artifactId>spring-cloud-starter-vault-config</artifactId>
</dependency>
<dependency>
   <groupId>org.springframework.cloud</groupId>
   <artifactId>spring-cloud-vault-config-databases</artifactId>
</dependency>
<dependency>
   <groupId>org.springframework.boot</groupId>
   <artifactId>spring-boot-starter-data-jpa</artifactId>
</dependency>
<dependency>
   <groupId>org.postgresql</groupId>
   <artifactId>postgresql</artifactId>
</dependency>

La seule chose que nous devons faire est de configurer l'intégration avec Vault via Spring Cloud Vault. Les paramètres de configuration suivants doivent être placés dans  bootstrap.yml (non  application.yml). Vous pouvez envisager d'exécuter cette application sur Nomad.

spring:
  cloud:
    vault:
      uri: http://localhost:8200
      token: ${VAULT_TOKEN}
      postgresql:
        enabled: true
        role: default
        backend: database
  datasource:
    url: jdbc:postgresql://localhost:5432/posgtres

La partie importante de la configuration visible ci-dessus se trouve sous la propriété  spring.cloud.vault.postgresql. Suite à la documentation de Spring Cloud "Le nom d'utilisateur et le mot de passe sont stockés dans  spring.datasource.username et  spring.datasource.password donc l'utilisation de Spring Boot récupérera les informations d'identification générées pour vous DataSourcesans autre configuration" . Pour plus de détails sur l'intégration entre Spring Cloud et le moteur de base de données Vault, vous pouvez vous référer à mon article Secure Spring Cloud Microservices with Vault and Nomad .

7. Soyez toujours à jour

Cette l'une des meilleures pratiques peut être appliquée n'importe où, pas seulement en règle générale pour la sécurité des microservices Spring. Nous utilisons généralement des bibliothèques open-source dans nos applications, il est donc important d'inclure les dernières versions de celles-ci. Ils peuvent contenir des mises à jour critiques pour les vulnérabilités divulguées publiquement contenues dans les dépendances d'un projet. Il existe également plusieurs scanners de dépendances comme Snyk ou OWASP .
